## 1.0.0의 정의 (Definition of Done — CLI)

> **5대 기능이 전부 실키·실파일로 검증되고, 공개 API가 동결(시맨틱 버저닝 약속)되며, CI 자동 발행·OTA 자동 업데이트가 동작하는 상태.**

### v0.4.0 — 한글 표·양식 정확 수정 (구조 보존)

> 테마: "글자는 고쳐진다"에서 "한글 문서답게 고쳐진다"로. 기능 ④(HWP/HWPX 수정)의 정확도를 실무 수준으로.

**현재 능력 실측(2026-06-16, kordoc 2.7.6):**

| 케이스 | 현재 | 근거 |
|---|---|---|
| 단순 격자 표 셀 편집 | ✅ 정확 | 3×3 왕복 + 셀 편집 무손실 |
| 라벨/값 양식(2열 표) 값 채우기 | ✅ 정확 | `extractFormFields` conf 1.0, 값 치환 반영 |
| 병합셀 표(colSpan/rowSpan) | ❌ **평면화** | 왕복 후 전 셀 span=1 — 병합 구조 파괴 |
| 진짜 양식 개체(입력상자·누름틀·콤보박스) | ❌ 미지원 | label-value 표 휴리스틱만 |
| 셀 서식(배경색·정렬·글꼴)·중첩표 | ⚠️ 손실 | markdown 왕복 한계 |

**근본 원인**: kordoc 쓰기 API가 `markdownToHwpx(markdown)` 하나뿐 — `IRBlock[]` 직접 쓰기 경로 없음. 모든 편집이 markdown 병목을 거쳐, markdown이 표현 못 하는 구조(병합·서식·양식 개체)가 손실됨. `templateArrayBuffer`는 파일 골격·스타일만 제공하고 본문 표는 markdown에서 재생성.

**실문서 검증(rhwp 샘플, 2026-06-16):** 실제 한글 문서는 병합셀로 가득하고 현재 편집 경로가 이를 파괴함을 실측.

| 파일 | 표 | 병합셀 | markdown 왕복 후 |
|---|---|---|---|
| table-text.hwpx | 1 | 2 | 병합 0 (손실), 표 유지 |
| table-vpos-01.hwpx | 10 | 19 | 병합 0, 표 10→6 (일부 소멸) |
| 기부·답례품 보고서_양식.hwpx | 55 | 81 | 병합 0, **표 55→2 (문서 파괴)** |

**제안 방향 — PoC 검증 완료 ✅**: markdown 전체 재생성 대신 **HWPX(zip+XML) 셀 단위 직접 패치**. PoC(table-text 셀 `72.5→999.9`): 대상 셀만 변경, 병합셀 2개·타 셀 전부 보존(diff 0). 복잡 양식(form-gibu, 표55·병합81) 항등 re-zip도 **완전 무손실**(JSZip, mimetype STORE 우선). → **XML 직접 패치 접근 확정**(외부 의존: jszip 직접 추가 예정).

**현재 결론**: **양사 모두 제3자 앱의 구독 OAuth 사용을 허용하지 않는다.** BYOK(API 키)가 유일한 정상 경로. 토큰은 1st-party 클라이언트 ID에 스코프되고, 정책은 하루아침에 바뀐다(2026-01 Anthropic 차단→환원, 2026-02 금지 명시, 2026-06 과금 분리). → v0.5.0은 **"벤더가 공식 개방하면 즉시 붙일 수 있게 준비"**가 현실적 목표이고, 실제 구독 사용은 공식 지원 전까지 **차단 항목**으로 둔다.

| # | 작업 | 완료 기준 |
|---|---|---|
| 0 | **타당성·약관 스파이크(필수 선행)** | 양사 공식 OAuth/구독-프로그래밍 지원 범위·약관·토큰 스코프 확정. 비공식 우회(토큰 추출·클라이언트 임퍼서네이션)는 명시적으로 채택 제외 |

### v0.6.0 — kordoc 갭 보완 (찾기/바꾸기·표 구조) — 자체 XML 패치로 해결

> 테마: kordoc이 못 하는 갭(찾기/바꾸기, 표 구조 편집)을 채운다. **당초 rhwp(WASM)로 시도했으나 rhwp가 복잡 문서를 손상시켜, 전부 자체 XML 패치로 재구현하고 rhwp를 제거**했다.
>
> **최종 결과(2026-06-16)**: ✅ 찾기/바꾸기·표 구조 편집을 **surgical XML 패치**로 구현 — 복잡 문서(중첩표·이미지)에서도 무손실, 한컴 검증. **`@rhwp/core` 의존 완전 제거**(`08051e8`, 사용자 5.6MB WASM 미설치). 모든 문서 편집이 kordoc 읽기 + 자체 XML 패치로 통일됨. rhwp의 강점(.hwp 직접 저장·시각 렌더)은 신뢰성(저장 손상)·범위(브라우저 전용) 한계로 채택하지 않음.

**타당성 실측(2026-06-16, `@rhwp/core` 0.7.15):**
- ✅ **Node 헤드리스 구동** — WASM이 브라우저 없이 `initSync`/`new HwpDocument(bytes)`. 렌더 메서드만 Canvas 전용(GUI용).
- ✅ **저장 가능** — `exportHwpx()` + **`exportHwp()`(.hwp 바이너리 쓰기, kordoc 불가)**. table-text identity export → 병합 보존·kordoc 재파싱·**한컴 렌더 확인**.
- ✅ 풍부한 편집 API — 표 행·열 삽입/삭제·`mergeTableCells`, `setFormValue`(셀 내 포함), `replaceOne`, `insertParagraph`, 중첩표 path API.

**kordoc이 못 하는데 rhwp가 채우는 갭**: ① `.hwp` 직접 저장 ② 표 구조 편집(행·열·병합) ③ 중첩표 접근 ④ 찾기/바꾸기 ⑤ 시각 렌더(SVG/HTML/PNG, GUI).

**리스크**: 5.6MB WASM 번들(콜드스타트·설치 크기); rhwp HWPX **저장 손상 이력**(트러블슈팅 다수·`exportHwpVerify`·과거 #196 저장 비활성화) → **기능별 한컴 검증 필수**; 순수 TS ↔ WASM 패러다임 혼재.

**원칙**: 현재 검증된 도구(`propose_cell_edit`·양식 개체 XML 패처)는 유지. rhwp는 **외부 의존(@rhwp/core)으로 추가**(번들 인라인 X, 런타임 lazy-init), 각 쓰기 기능은 한컴 렌더 검증 후 합류. 1.0 GA 비의존(점진 도입).

> **⚠️ 중대 실측(2026-06-16) — rhwp `exportHwpx`가 복잡 문서를 손상시킴**: 중첩표·이미지가 든 실문서(table-vpos-01)를 **편집 없이 항등 변환**해도 블록 38→33·표 10→9로 내용 손실, 한컴이 "손상/변조" 경고+빈 화면. kordoc는 손상 출력도 parse 성공이라 단순 게이트로 못 잡음. → **대응: `detectStructuralLoss` 게이트(`12353d4`)** — export 후 블록 타입별 개수 보존 확인, 감소 시 중단. 결과: **rhwp 쓰기 도구(#1·#2)는 "rhwp가 깨끗이 왕복하는 문서"에서만 동작하고, 복잡 문서(중첩표·이미지)에서는 안전하게 거부**한다. 즉 rhwp의 실용 범위는 단순~중간 복잡도 문서로 제한됨(복잡 문서 손상은 rhwp 상류 과제).
